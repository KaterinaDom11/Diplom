# План тестирования мобильного приложения «В Хосписе»

## 1. Общая информация
*   **Название приложения:** «В Хосписе»
*   **Версия:** 1.0.0
*   **Тип:** Нативное Android-приложение
*   **Сфера деятельности:** Социальная сеть / Социальное взаимодействие

## 2. Введение
Настоящий документ описывает методы, подходы и стратегии тестирования, которые будут применены для проверки работоспособности и качества нативного Android-приложения «В Хосписе».

### Цели тест-плана
*   Определить компоненты приложения, подлежащие тестированию.
*   Описать стратегии и типы тестирования.
*   Определить необходимые ресурсы (окружение, устройства, инструменты).
*   Зафиксировать результаты в виде отчетов и баг-репортов.

## 3. Рамки (Scope) проекта
В рамках тестирования будут проверены следующие ключевые функциональные блоки:
1.  **Авторизация:** Форма входа в систему.
2.  **Лента новостей:**
    *   Просмотр главной страницы "Новости".
    *   Просмотр страницы отдельной новости.
    *   Просмотр страницы "О приложении"
3.  **Цитаты:** Просмотр страницы "Цитаты".
4.  **Управление контентом (CRUD для новостей):**
    *   Добавление новой новости.
    *   Редактирование существующей новости.
    *   Удаление новости.
    *   Просмотр созданных новостей.

## 4. Подход и стратегия тестирования
### 4.1. Функциональное тестирование
**Цель:** Убедиться, что приложение работает в строгом соответствии с заявленными функциональными требованиями, а критичные сценарии использования не содержат ошибок. Это основная и наиболее объемная часть тестирования.

### 4.2. Основные типы тестирования
*   **Функциональное тестирование** (Functional Testing) — проверка работы функций.
*   **Тестирование пользовательского интерфейса (UI Testing)** — проверка соответствия макетам, отображения элементов.
*   **Юзабилити-тестирование (Usability Testing)** — оценка удобства и интуитивности интерфейса.
*   **Кросс-платформенное/кросс-устройство тестирование (Compatibility)** — проверка запуска и работы на различных эмуляторах и физических устройствах с поддерживаемыми версиями Android.

## 5. План работ
1.  Подготовка плана тестирования (текущий этап).
2.  Подготовка чек-листов (Checklists).
3.  Разработка тест-кейсов (Test Cases).
4.  Проведение ручного функционального тестирования и оформление баг-репортов.
5.  Проведение автоматизированного тестирования.
6.  Анализ результатов и подготовка финального отчета.
   
##  6. План автоматизации
### 6.1. Цели автоматизации
* Обеспечить стабильность работы критичного функционала (логирование, создание новостей, просмотр новостей).
* Автоматизировать рутинные проверки на разных конфигурациях устройств.
* Ускорить feedback-цикл для разработчиков за счет интеграции в CI/CD.
* Повысить уверенность в стабильности приложения перед релизом.
### 6.2. Инструментарий
*	Фреймворк для автоматизации: Appium (стандарт для кроссплатформенной автоматизации мобильных приложений).
*	Язык программирования: Java + JUnit.
*	Среда разработки: VS Code / IntelliJ IDEA.
*	Эмуляторы/Устройства: Android Studio Emulator (Pixel 4, API 30), реальное устройство HONOR X8b.
*	Система сборки: Gradle (для Android).
*	Система контроля версий: Git (репозиторий для кода автотестов).
*  CI/CD система: GitHub Actions / Jenkins (для настройки автоматических прогонов).
*  Система отчетности: Allure Report с интеграцией JUnit аннотаций (@Step, @Attachment).

### 6.4. Этапы автоматизации 
#### Этап 1. Подготовка инфраструктуры (1 день)
* Настройка проекта в IDE, конфигурация build.gradle, подключение зависимостей (Appium, JUnit, Allure).
* Создание базовых классов: BaseTest (инициализация драйвера, общие методы), TestData.* 
* Настройка эмулятора и подключение реального устройства для отладки.

#### Этап 2. Автоматизация критичных сценариев (2-3 день)
* Реализация Page Objects для ключевых экранов: LoginScreen, MainScreen, NewsScreen.
* Написание и отладка автотестов для сценариев AUTH-01, AUTH-02, NEWS-01, NEWS-02.
* Настройка генерации и визуализации Allure-отчетов после прогона.

#### Этап 3. Расширение покрытия и интеграция с CI (4 - 5 день)
* Реализация сценариев NEWS-03, NEWS-04, требующих сложной подготовки и очистки данных.
* Интеграция с CI/CD системой (например, GitHub Actions) для запуска тестов по git push или по расписанию.
* Документирование проекта автотестов.

#### 6.5. Критерии оценки результатов
* Критерий успешного выполнения автотеста: Все шаги сценария пройдены, фактический результат совпадает с ожидаемым, тест завершен без ошибок фреймворка или окружения.
* Критерий приемки этапа автоматизации: Успешно выполняется 95% автоматизированных тестов из запланированных для данного этапа.

**Ключевые метрики:**
* Стабильность (Stability Rate): ≥ 90% успешных прогонов тестовой сборки.
* Покрытие сценариев (Scenario Coverage): ≥ 70% критических пользовательских сценариев покрыты автотестами.
